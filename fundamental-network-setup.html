架构概览 已验证

这套架构解决的核心问题是：在中国大陆使用 Tailscale 组建内网 + 翻墙代理时，两者路由冲突导致开发工具（Claude Code 等）频繁断连。方案将 Tailscale 和代理的职责分离，统一通过 Tailscale WireGuard 隧道承载代理流量，消除冲突。

WSL / Claude Code │ proxy alias → <WINDOWS_TAILSCALE_IP>:<V2RAYN_PORT> ▼ Windows V2RayN (SOCKS/HTTP local proxy) │ VLESS Reality 协议 ▼ Tailscale WireGuard Tunnel (100.x.x.x mesh) │ ~170ms avg latency (CN → US) ▼ VPS - US Region (Xray VLESS Reality) │ ▼ Global Internet (Google, Claude API, GitHub) China Direct → Baidu, WeChat, Feishu (~0.15s, bypasses proxy)

核心价值：Tailscale 只管组网（设备互通），V2RayN 只管代理（翻墙），两者职责分离不再打架。WSL 中 Claude Code 稳定访问 Anthropic API。
ℹ
关键设计决策：Tailscale 不使用 exit node 模式做翻墙（exit node 会把所有流量都发到 VPS，国内流量反而变慢）。分流在客户端侧完成（V2RayN 路由规则），Tailscale 只提供加密隧道。

VPS 配置步骤 已验证

安装 Tailscale

▸

在 VPS 上通过 SSH 执行。适用于 CentOS / Ubuntu / Debian。

curl -fsSL https://tailscale.com/install.sh | sh

安装完成后启动并加入 tailnet，同时注册为 exit node（备用）：

tailscale up --advertise-exit-node --hostname=my-vps-proxy

浏览器打开终端返回的 login.tailscale.com 链接，用你的 Tailscale 账号授权。

ℹ
需要在 Tailscale Admin Console → Machines → 你的 VPS → Edit route settings 中手动批准 exit node。Tailscale Personal 免费计划支持 100 台设备。

通过 3X-UI 配置 Xray VLESS Reality 节点

▸

在 3X-UI 面板中：入站列表 → 添加入站。参考参数如下（按需调整）：

协议：VLESS
端口：自选（如 18889）
传输协议：TCP
安全性：Reality
uTLS：chrome
dest (目标)：www.microsoft.com:443
SNI：www.microsoft.com
Flow：建议设为 xtls-rprx-vision（新版 Xray 推荐）

面板会自动生成 UUID、公钥、私钥和 Short ID。记下公钥和 Short ID，客户端需要用。

⚠
不要在公开场合分享你的 UUID、私钥、分享链接。定期轮换 UUID 是好习惯。

修复 Xray 路由规则（关键！）

▸

Xray 默认配置会 block 所有私有 IP（包括 Tailscale 的 100.x.x.x 段），必须删除这条规则，否则通过 Tailscale 内网连入的代理流量会被静默丢弃。

# 查看 Xray 配置中的 geoip:private 规则位置
grep -n "private" /usr/local/x-ui/bin/config.json

# 查看上下文确认要删的行范围
sed -n 'START,ENDp' /usr/local/x-ui/bin/config.json

# 删除 geoip:private → blocked 那整段规则
# （行号以你实际的为准，示例：）
sed -i '19,25d' /usr/local/x-ui/bin/config.json

# 确认只剩 privateKey（不是路由规则的那个）
grep -n "private" /usr/local/x-ui/bin/config.json

# 重启 3X-UI + Xray
x-ui restart
⚠
这是最容易遗漏的坑。表现为：ping Tailscale IP 正常，curl TCP 端口也通（返回 400），但代理客户端始终超时。原因是 Xray 在 VLESS 握手后做路由判定，把 100.x.x.x 来源的流量 block 了。删除 geoip:private 规则即可。

WSL 代理配置 已验证

清理旧代理配置

▸

如果 WSL 之前配过 V2Ray / Clash 代理环境变量，需要先清理，避免冲突。

# 备份
cp ~/.bashrc ~/.bashrc.bak.$(date +%Y%m%d)

# 查看旧配置
grep -n "proxy\|PROXY\|10808\|10809\|WIN_HOST\|hostip" ~/.bashrc

# 按行号删除对应行（示例）
sed -i 'START,ENDd' ~/.bashrc

# 确认清理干净
grep -n "proxy\|PROXY\|10808\|hostip" ~/.bashrc

添加新代理配置

▸

WSL 通过 Windows 宿主机的 Tailscale IP 访问 V2RayN 的本地 SOCKS/HTTP 端口。流量路径：WSL → Windows V2RayN → Tailscale tunnel → VPS Xray → Internet。

# 添加到 ~/.bashrc 末尾
cat >> ~/.bashrc << 'EOF'

# === Proxy via Windows V2RayN (over Tailscale LAN) ===
export PROXY_HOST="<WINDOWS_TAILSCALE_IP>"
export PROXY_PORT="<V2RAYN_LOCAL_PORT>"
alias proxy="export https_proxy=http://$PROXY_HOST:$PROXY_PORT http_proxy=http://$PROXY_HOST:$PROXY_PORT all_proxy=socks5://$PROXY_HOST:$PROXY_PORT; echo '✅ 代理已开启 → V2RayN'"
alias unproxy="unset http_proxy https_proxy all_proxy; echo '❌ 代理已关闭'"
proxy
EOF

source ~/.bashrc
ℹ
末尾的 proxy 使每次开新终端自动开启代理。如需临时关闭，输入 unproxy。替换 <WINDOWS_TAILSCALE_IP> 为你 Windows 机器的 Tailscale IP（100.x.x.x），<V2RAYN_LOCAL_PORT> 为 V2RayN 的本地 SOCKS 端口（通常 10808）。

前提条件 checklist

▸

WSL 代理正常工作需要以下条件同时满足：

☐ Windows Tailscale 在线（系统托盘图标正常连接）

☐ Windows V2RayN 运行中

☐ V2RayN 设置中开启了"允许局域网连入"（Allow connections from LAN）

☐ V2RayN 代理模式选择"自动配置系统代理"（不要用 TUN 模式，会与 Tailscale 冲突）

☐ V2RayN 节点地址已改为 VPS 的 Tailscale IP（100.x.x.x），不是公网 IP

排障记录 经验教训

常见问题及解决方案

症状：代理客户端通过 Tailscale IP 连 VPS 的 Xray 端口，每次都超时。但 curl -k https://TAILSCALE_IP:PORT 返回 400（TCP 层通）。
原因：Xray 路由配置中 geoip:private → blocked。Tailscale 的 100.x.x.x 属于私有 IP 段，VLESS 握手完成后被 Xray 路由丢弃。
修复：删除 Xray 配置文件中的 geoip:private block 规则，重启 x-ui。详见"VPS 配置步骤"第 3 步。
症状：sing-box (1.11.x) 通过 VLESS Reality 连接 Xray 26.x 始终超时，但 V2RayN 连同一节点正常。
原因：sing-box 与新版 Xray 的 VLESS Reality 握手存在兼容性问题。Xray 26.x 警告"VLESS with no Flow is deprecated"。
方案：1) 改用 V2RayN 本地端口做中转（已验证可用）；2) VLESS 节点加 Flow: xtls-rprx-vision；3) VPS 端用 sing-box 替换 Xray（两端同一实现，兼容性最好）。
症状：sing-box 启动时 rule-set 下载失败（remote 模式）。
原因：rule-set 从 GitHub 下载需要翻墙，但代理还没启动——鸡生蛋问题。
修复：改用 local 模式。先通过其他渠道下载 .srs 文件到本地，配置中 type 改为 "local"，指定 path。
症状：sing-box 1.8+ 启动报错 "geoip database is deprecated"。
原因：sing-box 1.8 起废弃了旧版 geoip/geosite 格式。
修复：配置文件的 route.rules 不再用 "geoip": ["cn"]，改用 "rule_set": "geoip-cn" + route.rule_set 定义。
症状：V2RayN 翻墙时灵时不灵，WebRTC leak 测试显示真实 IP。
原因：V2RayN 的"系统代理"被清除了。TUN 模式与 Tailscale 路由冲突。WebRTC 会绕过系统代理直连。
修复：V2RayN 托盘菜单选"自动配置系统代理"。不要开 TUN 模式。浏览器装 WebRTC Leak Shield 插件。
症状：WSL 中 Tailscale + V2Ray 双网络栈导致 Claude Code 频繁卡死。
原因：WSL 内部同时运行 Tailscale daemon + 配置了指向 Windows V2Ray 端口的代理，两层网络栈抢路由。
修复：本架构的核心——职责分离。Tailscale 只管组网（设备互通），代理只走 V2RayN 本地端口。WSL 内部的 Tailscale 不参与代理流量。

sing-box 方案 备选 / 待修复

sing-box 可在 WSL 内独立运行代理客户端，不依赖 Windows V2RayN。链路更短（WSL → Tailscale → VPS，少一跳），但需要解决与 Xray 的 Reality 兼容性问题。

# 安装 sing-box (Linux amd64)
curl -Lo /tmp/sing-box.tar.gz \
  https://github.com/SagerNet/sing-box/releases/download/v1.11.4/sing-box-1.11.4-linux-amd64.tar.gz
tar -xzf /tmp/sing-box.tar.gz -C /tmp/
sudo cp /tmp/sing-box-1.11.4-linux-amd64/sing-box /usr/local/bin/
sudo chmod +x /usr/local/bin/sing-box

# 配置文件位置
~/.config/sing-box/config.json

# 下载 rule-set 到本地（解决鸡生蛋问题）
mkdir -p ~/.config/sing-box/ruleset
curl -L -o ~/.config/sing-box/ruleset/geosite-cn.srs \
  https://raw.githubusercontent.com/SagerNet/sing-geosite/rule-set/geosite-cn.srs
curl -L -o ~/.config/sing-box/ruleset/geoip-cn.srs \
  https://raw.githubusercontent.com/SagerNet/sing-geoip/rule-set/geoip-cn.srs

# 启动
sing-box check -c ~/.config/sing-box/config.json
sing-box run -c ~/.config/sing-box/config.json &

# 本地代理端口：127.0.0.1:7890 (HTTP/SOCKS mixed)

sing-box 配置文件模板（VLESS Reality + 本地分流）：

{
  "log": { "level": "warn" },
  "inbounds": [{
    "type": "mixed",
    "listen": "127.0.0.1",
    "listen_port": 7890
  }],
  "outbounds": [
    {
      "type": "vless",
      "tag": "proxy",
      "server": "<VPS_TAILSCALE_IP>",
      "server_port": <XRAY_PORT>,
      "uuid": "<YOUR_UUID>",
      "tls": {
        "enabled": true,
        "server_name": "www.microsoft.com",
        "utls": { "enabled": true, "fingerprint": "chrome" },
        "reality": {
          "enabled": true,
          "public_key": "<YOUR_PUBLIC_KEY>",
          "short_id": "<YOUR_SHORT_ID>"
        }
      }
    },
    { "type": "direct", "tag": "direct" }
  ],
  "route": {
    "rule_set": [
      { "type": "local", "tag": "geosite-cn", "format": "binary",
        "path": "/home/YOU/.config/sing-box/ruleset/geosite-cn.srs" },
      { "type": "local", "tag": "geoip-cn", "format": "binary",
        "path": "/home/YOU/.config/sing-box/ruleset/geoip-cn.srs" }
    ],
    "rules": [
      { "ip_is_private": true, "outbound": "direct" },
      { "rule_set": "geosite-cn", "outbound": "direct" },
      { "rule_set": "geoip-cn", "outbound": "direct" }
    ],
    "final": "proxy",
    "auto_detect_interface": true
  }
}

修复方向：1) VPS 端用 sing-box 服务端替换 Xray；2) VLESS 加 Flow: xtls-rprx-vision；3) VPS 额外开 SOCKS5 端口（纯转发，不走 Reality）。

V2RayN 分流规则优化

▸

在 V2RayN 的路由设置中导入 GeoIP CN 和 GeoSite CN 规则，确保国内域名/IP 走直连、被墙域名走代理。

主备 VPS 节点

▸

第二台 VPS 按同样步骤配置：Tailscale + Xray + 删 geoip:private。在 V2RayN 中添加两个节点，主节点故障时一键切换。

手机 / MacBook 代理方案

▸

方案 1：Tailscale + 本地客户端。设备上装 Tailscale + Shadowrocket/V2Box/Clash，节点地址填 VPS Tailscale IP。不需要中转服务。

方案 2：中转 + 公网直连。使用 BGP 中转入口连 VPS 公网 IP。延迟更低但多一层依赖。

VPS 角色分工

▸

VPS 1：主代理节点

VPS 2：备用代理节点

VPS 3：轻量服务（自动化 cron、监控、bot）— 不跑代理

VPS 4：按需启停，节省成本

修复 sing-box 直连方案

▸

解决兼容性后，WSL 可脱离 V2RayN 依赖，链路更短更稳。优先级不高，当前方案够用。
