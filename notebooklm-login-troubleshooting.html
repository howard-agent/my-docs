通过 Chrome 映射的 NotebookLM Web App 突然注销，重新打开后没有登录按钮，页面卡在产品介绍页，URL 尾部强制带有 ?location=unsupported 参数。手机端同步测试时提示 Could not load NotebookLM。

🔍 核心原因分析 ROOT CAUSE

经过多轮排查，导致 NotebookLM 判定为"不支持地区"的原因主要有三个层级：

层级 原因 说明
01 节点出口 IP 不合规 初期使用"广港 IEPL"节点，出口 IP 落在香港，而 NotebookLM 目前不支持香港地区。
02 WebRTC 真实 IP 泄露（最隐蔽） 即使代理 IP 切换到了美国，桌面浏览器依然通过底层 WebRTC 协议向 Google 暴露了真实的 [公网IP]（带有中国标识），导致伪装失效。
03 DNS 污染与缓存死锁 URL 携带的 location=unsupported 会被浏览器缓存，且部分 DNS 解析未走代理通道（如直连了 223.5.5.5），导致后续验证依然失败。

🛠️ 最终解决方案与操作步骤 SOLVED

更换合规的节点 IP

▶

放弃使用香港或特征明显的国内中转节点。

切换至 美国 (US)、新加坡 (SG) 或 日本 (JP) 等原生支持地区的节点。

彻底封堵 WebRTC 泄露（关键步骤）

▶

在不开启全局 TUN 模式的情况下，必须从浏览器层面阻止真实 IP 泄露。

  • 1 在 Chrome 应用商店安装 WebRTC Leak Prevent 插件。
  • 2 进入插件选项，将 WebRTC IP handling policy 设置为 Disable non-proxied UDP (force proxy)。默认选项无法完全隐藏公网 IP。
  • 3 访问 browserleaks.com/webrtc 验证，确保 Public IP Address 一栏显示为空白 (-) 或仅显示代理服务器的美国 IP。

清除浏览器"错误记忆"

▶
  • 1 彻底关闭所有 Chrome 窗口（包含无痕模式）以及映射生成的本地 Web App 窗口。
  • 2 按 Command + Shift + Delete 清除近期的 Cookie 和缓存。
  • 3 在浏览器地址栏手动输入纯净网址 https://notebooklm.google.com/，切勿直接刷新带有 unsupported 尾缀的历史页面。
  • 4 成功看到 "Try NotebookLM" 并登录后，可重新生成桌面 Web App（PWA）。
✅
按以上步骤操作后，NotebookLM 登录页面应正常显示，无 ?location=unsupported 参数。

⚠️ TUN 模式与 DNS 死锁避坑 PITFALL

在尝试解决网络问题时，曾尝试开启 v2rayN 的 TUN 模式，但引发了大量 i/o timeout 报错（如无法连接阿里 DNS 223.5.5.5:443）。

根本原因
路由环路（Routing Loop）
TUN 模式强制接管所有网卡流量，代理工具在尝试通过直连网络解析节点域名或国内 DNS 时，流量又被绕回 TUN 虚拟网卡，形成死锁。
建议结论
不推荐强制开启 TUN
如果系统代理 + WebRTC 插件已能让网页版顺畅运行，无需开启 TUN 模式，以避免复杂的底层网络冲突。
⚠️
TUN 模式适合需要代理全局流量（含非浏览器应用）的场景，但对于单纯解决 NotebookLM 访问问题，系统代理 + WebRTC 封堵已足够，引入 TUN 得不偿失。

💻 Windows 环境补充建议 WSL

针对 Windows + WSL 用户，若 Windows 环境下 NotebookLM 一切正常，但 WebRTC 仍有泄露风险：

现状风险已建立 Session，Google 暂未阻断，但属于"定时炸弹"——未来触发深度检测可能再次掉线。
处理建议在 Windows 的桌面浏览器中同样安装并严格配置 WebRTC 防泄露插件。
WSL 内部命令行操作不受浏览器 WebRTC 影响，无需特殊处理。
✅
WSL 内的终端/CLI 工具走的是系统代理或直连，与浏览器 WebRTC 泄露问题完全独立，不需要额外配置。
