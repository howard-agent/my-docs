🔌 核心大模型通道 (Providers) 接入指南 PROVIDER

⚠️
OpenClaw 3.8 的 Provider 校验机制极其严格（强迫症级别），配置时必须保证字段完整性，缺少任何必要字段（如 baseUrl、models 数组）都会实时报错阻断。

AIHubMix — 官方插件自动化接入

▶

AIHubMix 提供了专用的认证插件，可自动同步分类所有模型（涵盖 Claude、Gemini、OpenAI 等），是最省心的接入方式。

安装插件：

openclaw plugins install @akakneles/aihubmix-auth

重启容器（宿主机执行）：

sudo docker restart openclaw

交互式认证（按提示输入 API Key，插件自动配置路由和 Header）：

openclaw models auth login --provider aihubmix --method api-key --set-default

OpenRouter — 系统原生免驱接入

▶

OpenRouter 在 3.8 版本中作为原生 Built-in 支持，不需要使用 plugin login 命令。

配置方式：通过修改配置表或使用以下命令激活：

openclaw config set auth.openrouter "[密钥]"

调用时直接在模型 ID 前加前缀即可，例如：

openrouter/anthropic/claude-3-opus

VectorEngine — 极其严格的 CLI 写入踩坑记录

▶

VectorEngine 作为 OpenAI 兼容接口，尝试使用 openclaw config set 逐行写入时，遭遇了实时完整性校验拦截。

❌ 问题根因：逐行写入时，每次写入后 OpenClaw 立即校验，缺少 baseUrl、缺少 models 数组等都会直接报错阻断。

最终解决方案（复合 JSON 一次性注入）：必须将完整对象包含 api 类型、URL、Key 以及 Models 数组一次性写入：

openclaw config set models.providers.vectorengine '{
  "api": "openai-completions",
  "baseUrl": "https://api.vectorengine.ai/v1",
  "apiKey": "[密钥]",
  "models": [
    { "id": "deepseek-v3", "name": "DeepSeek V3" },
    { "id": "minimax-abab6.5-chat", "name": "MiniMax" }
  ]
}'

🧑‍💼 智能体 (Agent) 的配置与指派 DONE

当 Provider（管道）接通后，需要创建具体的 Agent（角色）并指派对应的 Provider 和 Model。

基础命令规范

openclaw config set agents.<角色名>.provider "<对应Provider名称>"
openclaw config set agents.<角色名>.model "<对应模型ID>"

实战举例（配置 worker 分身）

openclaw config set agents.worker.provider "vectorengine"
openclaw config set agents.worker.model "deepseek-v3"
✅
配置完成后重启容器，在飞书内通过 /worker <指令> 即可调用该分身。

🌐 网络与外网 Web UI 访问排查手册 TROUBLESHOOT

在尝试访问 http://[内网IP]:18789/?token=... 的 Web GUI 时遇到了各种网络问题，以下是排查经验。

宽带与 DDNS 限制 — ERR_CONNECTION_CLOSED

▶
现象使用腾讯云 DDNS 域名访问报 ERR_CONNECTION_CLOSED
根因家宽无公网 IPv4，仅配置了 IPv6 DDNS
当外网设备（如手机移动网络或部分公司 Wi-Fi）不支持 IPv6 时，域名解析直接失效，连接被关闭。

Tailscale + v2rayN 冲突处理

▶

为实现"随处访问"，启用了 NAS 上的 Tailscale 内网穿透。

冲突点Mac 端的 v2rayN 代理规则会劫持/丢弃 Tailscale 的虚拟局域网流量
解决方案在 v2rayN 中开启 Enable Tun（虚拟网卡模式）
开启 Tun 模式后，系统级路由可正确接管科学上网和 Tailscale 的 100.x.x.x 流量，两者不再冲突。

终极 503 报错 — HTTP ERROR 503

▶

网络打通后，通过 Tailscale IP 访问依然报 503，有两个根因：

原因 1容器内的 OpenClaw Gateway 默认只监听 127.0.0.1，拒绝外部（含 Tailscale 虚拟网卡）访问
# Docker 环境变量中补充以下配置
HOST=0.0.0.0
原因 2频繁修改配置导致 Gateway 进程假死
openclaw gateway stop
openclaw gateway start
💡
由于 Web UI 持续 503，后续果断转向纯 CLI 命令行完成所有配置，反而更高效稳定。
